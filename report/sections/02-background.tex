\section{Background and Related Work}
\label{sec:background}

\subsection{Horizontal Pod Autoscaler}
\label{subsec:bg-hpa}
The Kubernetes HPA~\cite{kubernetes2024hpa} periodically computes the desired number of replicas for a workload. By default, the controller runs every 15~s and obtains resource metrics from \texttt{metrics-server}. For a metric-based target, the simplified control rule is
\begin{equation}
  \label{eq:hpa}
  n_\mathrm{desired} = \left\lceil n_\mathrm{current} \cdot
  \frac{m_\mathrm{current}}{m_\mathrm{target}} \right\rceil,
\end{equation}
where the final result is clamped between \texttt{minReplicas} and \texttt{maxReplicas}. A \texttt{behavior} block can further limit how quickly the HPA scales up or down. For example, a scale-down stabilisation window prevents a short metric dip from immediately removing pods, and a percentage policy limits how many pods can be removed in one period.

When the metric is CPU utilisation, Kubernetes computes usage relative to \texttt{requests.cpu}. This design is sensible for CPU-bound synchronous services, but it is less informative for a queue consumer. A pod with \texttt{requests.cpu}=100~m and \texttt{limits.cpu}=300~m can report approximately 300\% utilisation whenever it is continuously busy. The HPA can see that the pod is busy, but it cannot distinguish a queue containing 5,000 messages from a queue containing 200 messages.

\subsection{Kubernetes Event-Driven Autoscaling}
\label{subsec:bg-keda}
KEDA~\cite{keda2024} extends Kubernetes through the \texttt{ScaledObject} custom resource. It provides scalers for message brokers, databases, HTTP endpoints, and other event sources~\cite{keda2024scalers}. For RabbitMQ, the scaler can read queue length and expose it to Kubernetes as an external metric. KEDA then creates and manages an HPA behind the scenes, so the Kubernetes scaling mechanism remains familiar while the metric becomes workload-aware.

For a RabbitMQ \texttt{QueueLength} trigger, the operator specifies a target number of queued messages per replica, denoted as $T$. If the current queue depth is $Q$, the intuitive desired replica count is
\begin{equation}
  \label{eq:keda}
  n_\mathrm{desired} \approx \left\lceil \frac{Q}{T} \right\rceil .
\end{equation}
This signal is closer to the service-level problem: the autoscaler responds to the amount of unfinished work rather than to the CPU symptoms of processing that work.

\subsection{Related Work}
\label{subsec:bg-related}
Autoscaling has been studied through taxonomies of reactive and predictive cloud scaling algorithms~\cite{qu2018taxonomy}, Kubernetes performance models~\cite{medel2016modelling}, HPA-focused experiments~\cite{nguyen2020hpa}, and large-scale serverless workload studies~\cite{shahrad2020serverless}. Our study has a narrower but controlled goal. We do not compare many algorithms. Instead, we hold the application, broker, hardware, resource limits, replica bounds, HPA behaviour settings, and load pattern constant, so that the measured difference can be attributed to the scaling signal itself.

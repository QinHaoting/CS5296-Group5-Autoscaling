\begin{abstract}
Queue-backed microservices are common in cloud systems, but they are difficult to scale with a CPU-only signal. A growing message queue represents unfinished work before the consumer's CPU history fully reflects it, and a CPU-limited pod can remain ``busy'' even after the incoming burst has already stopped. This project implements a reproducible technical evaluation of two Kubernetes autoscaling strategies for the same RabbitMQ-backed Spring Boot consumer: the standard Horizontal Pod Autoscaler (HPA), which scales from CPU utilisation, and Kubernetes Event-Driven Autoscaling (KEDA), which scales from RabbitMQ queue length. We deploy the system on a single-node K3s cluster running on an AWS EC2 \texttt{m5.large} instance, generate a 10-second flash-crowd workload targeting 1,000 messages/s, and run three trials per strategy. All application code, resource limits, replica bounds, and HPA \texttt{behavior} parameters are kept identical; only the scaling signal differs. In our measurements, KEDA reacts 10.4~s faster on average than the CPU-based HPA (63.4~s vs. 73.8~s), reaches peak capacity about 10~s earlier, drains the queue 12.2~s sooner, and begins scaling down during the observation window while HPA remains at six pods. The results show that, for event-driven queue consumers, using backlog as the autoscaling signal provides clearer and earlier information than CPU tuning alone.
\end{abstract}

\section{Experimental Setup}
\label{sec:setup}

\subsection{Environment}
\label{subsec:env}
The experiment runs on AWS EC2 in \texttt{us-west-2}. The node is an \texttt{m5.large} instance with 2~vCPU and 8~GiB RAM, running Ubuntu~22.04~LTS. K3s v1.30 provides the Kubernetes cluster, with Helm~3, \texttt{metrics-server} v0.7, and KEDA v2.13 installed by the setup scripts. RabbitMQ is deployed from repository-owned Kubernetes manifests using \texttt{rabbitmq:3.12-management}. Its readiness and liveness probes use TCP checks on port~5672, which avoids false HTTP-management failures on the constrained single-node cluster.

\subsection{Workload}
\label{subsec:workload}
The main load pattern is \texttt{load-test/patterns/burst.yaml}. It has three phases: 10~s of warm-up at 5~messages/s, 10~s of burst traffic targeting 1,000~messages/s, and 180~s of silence. The harness script also starts the collector before the producer and keeps observing after the producer exits, so each trial lasts about 390~s and captures roughly 355--360~s after the burst ends.

The producer is implemented in \texttt{load-test/producer.py} using the \texttt{pika} AMQP client~\cite{pika2024}. Each message contains a small JSON payload padded to 256~bytes. In practice, synchronous \texttt{basic\_publish} reaches around 740~messages/s on the test setup, so the realised burst contains approximately 7,100--7,500 messages rather than exactly 10,000. This ceiling affects both strategies equally because the same producer, pattern file, and broker path are used in all trials.

\subsection{Metrics}
\label{subsec:metrics}
The collector emits one CSV row per second with eight primary fields: timestamp, total pods, ready pods, queue depth, publish rate, delivery rate, cumulative delivered messages, and cumulative published messages. The analysis script then derives three groups of metrics:
\begin{itemize}
  \item \textbf{Responsiveness}: reaction latency from burst start to the first additional Ready pod, time to peak pod count, and queue-drain time from burst end to \texttt{queue\_depth}=0.
  \item \textbf{User-visible effect}: average delivery throughput during the drain period and total delivered messages per trial.
  \item \textbf{Resource use}: time to the first scale-down event, final number of Ready pods, peak pod count, and pod-seconds overshoot above the one-replica baseline.
\end{itemize}

\subsection{Trial Protocol}
\label{subsec:protocol}
We run three independent trials for each strategy and report the sample mean plus one sample standard deviation. Before every trial, the target Deployment is reset to one replica and the corresponding RabbitMQ queue is purged. The formal sequence is therefore: reset, purge, collect baseline, inject the burst pattern, observe the drain, and record any scale-down event. All six formal trials were executed sequentially on 2026-04-20, and the committed CSV files under \texttt{results/raw/} are the data source for the statistics and figures in this report.

\section{System Design}
\label{sec:design}

\subsection{Architecture Overview}
\label{subsec:arch}
Fig.~\ref{fig:arch} summarizes the implemented system. A single AWS EC2 \texttt{m5.large} instance (2~vCPU, 8~GiB RAM) hosts a lightweight K3s~\cite{rancher2024k3s} v1.30 cluster. The cluster contains three application namespaces: \texttt{rabbitmq} for the shared message broker, \texttt{baseline} for the CPU-based HPA experiment, and \texttt{keda} for the queue-length-based KEDA experiment. RabbitMQ exposes two durable queues, \texttt{baseline-queue} and \texttt{keda-queue}, so that each strategy is tested with an isolated backlog while sharing the same broker installation.

A Python producer running from the evaluator's machine sends AMQP messages to either queue. During each trial, \texttt{scripts/collect-metrics.sh} polls Kubernetes and the RabbitMQ Management API~\cite{broadcom2024rabbitmq} at approximately 1~Hz. It records pod counts, pod readiness, queue depth, ingress/egress rates, and cumulative published/delivered messages into CSV files. The same collection script is used for both strategies.

\subsection{Consumer Application}
\label{subsec:consumer}
The consumer is a Spring Boot~\cite{vmware2024springboot} application packaged as the shared image \path{htasteqin/cs5296-consumer:v1.0}. The main processing logic is in \texttt{MessageConsumer.java}. Each message is received through a Spring AMQP \texttt{@RabbitListener}, and the handler performs about 200~ms of CPU-bound busy work before acknowledging progress through Micrometer counters and timers.

This artificial CPU-bound work is intentional. If the handler used \texttt{Thread.sleep}, the queue could grow while CPU usage stayed low, making the CPU-based HPA fail for a trivial reason. By using busy work, the baseline HPA is given a real CPU signal, while KEDA is given the queue signal. The consumer pods request 100~m CPU and 256~MiB memory and are limited to 300~m CPU and 512~MiB memory. The same values appear in both the \texttt{baseline} and \texttt{keda} Deployment manifests.

\subsection{Scaling Strategies}
\label{subsec:strategies}
The two strategies share the same replica range and control limits. Both start with one replica and can scale to at most six replicas, a bound chosen to fit the two-vCPU single-node environment without destabilizing RabbitMQ. Both also use the same HPA \texttt{behavior} configuration:
\begin{itemize}
  \item \textbf{Scale-up}: no stabilisation window; maximum of \texttt{Percent 100\%/15s} and \texttt{Pods 4/15s}.
  \item \textbf{Scale-down}: 60~s stabilisation window; \texttt{Percent 50\%/60s} removal policy.
\end{itemize}
Only the metric differs. The baseline HPA uses CPU utilisation with a 50\% target. KEDA uses a RabbitMQ \texttt{QueueLength} trigger with target $T=100$ messages per replica, a 5~s polling interval, and a 60~s cooldown period.

\subsection{Controlled Variables}
\label{subsec:controls}
The experiment controls every major non-signal variable: the same Docker image, the same Spring Boot code path, the same resource requests and limits, the same per-message processing cost, the same RabbitMQ broker, the same EC2 node, the same burst traffic pattern, and the same metric-collection pipeline. This design prevents a common benchmarking mistake: comparing two autoscalers while also changing the workload, hardware, or application implementation.
